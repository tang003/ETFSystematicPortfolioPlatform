# 03 数据库设计

数据库：PostgreSQL

默认库名：`quant_etf`

ORM：SQLAlchemy

迁移工具：Alembic

## 迁移约定

- 初始 SQL 位于 `sql/init_schema.sql` 和 `sql/seed_assets.sql`，只负责首次创建 volume 时的初始化。
- 代码版本更新后的表结构变化必须使用 Alembic。
- API 和 worker 容器启动时会先执行 `alembic upgrade head`。
- 新增表、字段、索引时必须同步更新本文档和 `docs/09_开发日志.md`。

## 核心表

### app_user

用途：保存可登录系统的数据库用户。环境变量管理员账号仍保留为兜底账号。

关键字段：

- `username`：用户名，唯一。
- `password_hash`：密码哈希，只保存 PBKDF2 结果，不保存明文。
- `role`：角色，当前支持 `admin`、`researcher`、`viewer`。
- `display_name`：显示名称。
- `is_active`：是否启用；停用用户不能登录。
- `last_login_at`：最近登录时间。
- `created_at` / `updated_at`：创建和更新时间。

说明：

- 当前第一版只建立账号体系和角色字段。
- 业务数据按用户隔离仍是后续工作。
- `.env.production` 管理员账号不写入该表，继续作为兜底登录方式。

### audit_log

用途：记录变更类 API 操作审计。

关键字段：

- `actor_username`：操作者用户名。
- `actor_role`：操作者角色。
- `method`：HTTP 方法。
- `path`：请求路径。
- `action`：系统归类后的动作名。
- `status_code`：响应状态码。
- `duration_ms`：请求耗时。
- `client_ip`：客户端 IP。
- `request_id`：请求 ID。
- `detail`：安全摘要，不保存请求正文和密钥。
- `created_at`：记录时间。

说明：

- 当前只记录 POST、PUT、PATCH、DELETE 变更类请求。
- 登录接口暂不纳入通用审计中间件，后续可单独设计登录安全审计。

### asset_master

用途：ETF 主数据和研究池。

关键字段：

- `symbol`：ETF 代码，唯一。
- `name`：名称。
- `exchange`：交易所，例如 `SH`、`SZ`。
- `asset_class`：资产类别，例如 `equity`、`bond`、`gold`、`cash`、`qdii`。
- `asset_region`：区域，例如 `CN`、`HK`、`US`、`GLOBAL`。
- `currency`：币种，默认 `CNY`。
- `is_cross_border`：是否跨境 ETF。
- `is_leveraged`：是否杠杆。
- `is_inverse`：是否反向。
- `enabled`：是否进入研究池。
- `risk_level`：风险等级，1 最低，5 最高。
- `fund_company`：基金公司。
- `tracking_index`：跟踪指数。
- `listing_date`：上市日期。
- `fund_size`：基金规模，单位为元。
- `management_fee`：管理费率，小数格式，例如 0.005 表示 0.5%。
- `custody_fee`：托管费率。
- `expense_ratio`：综合费率。
- `tracking_error`：跟踪误差。
- `latest_premium_rate`：最新溢价率。
- `description`：说明。

说明：

- 全市场基础池当前正式通过 Tushare `fund_basic` 同步；`source=auto` 在正式口径下等同优先使用 Tushare。
- AKShare / 东方财富旧同步函数仍可能保留在代码中，但业务入口已停用，不应写入正式清洗行情。
- 新同步 ETF 默认不启用研究，避免全市场都进入策略。
- 当前持仓里输入新代码时，系统可以自动登记候选 ETF。

### trading_calendar

用途：交易日历。

关键字段：

- `trade_date`
- `market`
- `is_open`

用于判断行情缺失、同步范围和调仓日期。

### market_data_raw

唯一约束：`symbol + trade_date + source`

关键字段：

- `symbol`
- `trade_date`
- `open`
- `high`
- `low`
- `close`
- `volume`
- `amount`
- `source`
- `raw_payload`

### market_data_clean

用途：保存清洗后的行情，是策略、因子、回测、ETF 详情的主要行情来源。

唯一约束：`symbol + trade_date`

关键字段：

- `symbol`
- `trade_date`
- `open`
- `high`
- `low`
- `close`
- `volume`
- `amount`
- `is_adjusted`
- `data_status`

### etf_nav_premium

用途：保存 ETF 净值、市场价和溢价率。

当前是预留和部分能力，后续做跨境 ETF、LOF、QDII 风控时会更重要。

### data_quality_log

用途：记录数据同步、清洗、质量检查的结果。

常见类型：

- `missing_data`
- `duplicate_data`
- `zero_price`
- `zero_volume`
- `abnormal_return`
- `premium_missing`
- `cleaning_success`
- `market_sync_failed`

### factor_daily

用途：保存每日因子和 Alpha 综合评分。

关键字段：

- 均线：`ma20`、`ma60`、`ma120`、`ma200`
- 收益：`ret_20d`、`ret_60d`、`ret_120d`、`ret_250d`
- 波动：`volatility_60d`、`volatility_120d`
- 回撤：`drawdown_120d`、`drawdown_250d`
- 流动性：`liquidity_20d`
- 溢价：`premium_rate`
- 评分：`trend_score`、`momentum_score`、`volatility_score`、`drawdown_score`、`liquidity_score`、`premium_score`、`alpha_score`

### strategy_config

用途：保存策略参数。

当前核心策略：`core_etf_rotation`

### strategy_run

用途：记录每次策略运行。

策略结果都应能追溯到 `run_id`。

### data_source_config

用途：保存外部数据源配置，用于数据源管理页和运行时接口覆盖。

关键字段：

- `provider_code`：数据源代码，唯一，例如 `tushare`、`deepseek`。
- `provider_name`：数据源名称。
- `provider_type`：数据源类型，例如 `market`、`ai`、`news`、`broker`。
- `enabled`：是否启用。
- `base_url`：API 地址。
- `auth_type`：鉴权方式，例如 `token`、`bearer`、`none`。
- `secret_value`：后端密钥字段，前端不回显明文。
- `request_interval_seconds`：请求间隔。
- `quota_per_minute`：每分钟限额。
- `quota_per_day`：每日限额。
- `supported_usages`：支持用途，例如 `market_daily`、`asset_profile`、`ai_research`。
- `adapter_status`：业务适配状态，`runtime_supported` 表示已接入运行时，`metadata_only` 表示仅登记配置。
- `notes`：备注说明。

说明：

- `tushare` 的数据库 `base_url` 和 `secret_value` 会覆盖环境变量中的 `TUSHARE_API_URL` 和 `TUSHARE_TOKEN`。
- `deepseek` 的数据库 `base_url` 和 `secret_value` 会覆盖环境变量中的 `DEEPSEEK_BASE_URL` 和 `DEEPSEEK_API_KEY`。
- 新增未知数据源不会自动进入正式行情同步，必须先开发适配器。

### news_article

用途：保存财经新闻资讯，用于 ETF 关键词关联、AI 投研事件输入和后续每日简报。

唯一约束：`source + external_id`

关键字段：

- `source`：新闻数据源，例如 `juhe_finance_news`。
- `external_id`：外部新闻 ID。
- `title`：标题。
- `source_name`：原始新闻来源。
- `url`：新闻链接。
- `image_url`：图片链接。
- `publish_time`：发布时间。
- `summary`：摘要，后续可由 AI 生成。
- `content`：正文，当前聚合数据第一版通常不保存正文。
- `keywords`：关键词。
- `related_symbols`：关联 ETF 代码。
- `related_asset_class`：关联资产类别，例如 equity、bond、gold。
- `related_region`：关联区域，例如 CN、US、HK。
- `sentiment_score`：情绪分数，当前预留。
- `impact_level`：影响等级。
- `raw_payload`：原始响应。

说明：

- 第一版接入聚合数据财经新闻 API。
- 同步方式是后端批量入库，前端只读本地数据库。
- 关联 ETF 当前基于标题关键词、ETF 代码、ETF 名称和跟踪指数做初步匹配。
- 新闻只作为投研解释层，不直接构成买卖信号。

### agent_analysis_log

用途：保存 AI 投研委员会历史记录。

新增版本：`v0.39.0-agent-analysis-history`

关键字段：

- `symbol`：ETF 代码。
- `name`：ETF 名称。
- `start_date` / `end_date`：分析区间。
- `data_status`：数据状态，例如 `ready` 或 `missing_market_data`。
- `llm_used`：是否实际使用 DeepSeek。
- `llm_model`：模型名。
- `final_action`：最终建议。
- `final_score`：综合评分。
- `final_summary`：最终总结。
- `manager_commentary`：复合经理说明。
- `agents_payload`：各 Agent 的观点、证据、风险、建议。
- `warnings_payload`：数据缺失或限制提示。
- `created_at`：生成时间。

说明：

- 不保存 DeepSeek API Key。
- 每次运行 AI 投研都会新增一条记录。
- 前端 AI 投研页会读取最近历史记录。

## v0.65.8 多用户 owner 字段规范

本版开始区分“共享研究数据”和“用户私有业务数据”。

共享研究数据仍全局共用，包括 `asset_master`、`market_data_raw`、`market_data_clean`、`factor_daily`、`strategy_run`、`alpha_signal`、`target_portfolio` 等。这些数据来自数据源和策略计算，不按用户拆分。

用户私有业务数据新增 `owner_username` 字段，包括：

- `portfolio_position`：当前持仓快照。
- `holding_analysis_result`：持仓分析结果。
- `investment_plan`：定投计划。
- `investment_plan_suggestion`：定投建议。
- `report_log`：报告记录。
- `agent_analysis_log`：AI 投研历史记录。

迁移文件：`backend/alembic/versions/20260721_0011_user_owned_portfolio.py`。

约束变化：

- `portfolio_position` 旧唯一约束 `(position_date, symbol)` 改为 `(owner_username, position_date, symbol)`。
- 这样不同用户可以在同一天录入同一只 ETF，而互不覆盖。

兼容策略：

- 新写入数据必须设置 `owner_username`。
- 管理员读取时兼容旧版 `owner_username IS NULL` 历史数据。
- 普通数据库用户只能读取 `owner_username = 当前用户名` 的数据。
