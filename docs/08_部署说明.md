# 08 部署说明

## 环境要求

- Docker
- Docker Compose
- Python 3.11+

## Docker 启动

以下命令是本地开发启动方式：

在项目根目录执行：

```bash
cp .env.example .env
docker compose up --build
```

如果要启用 Tushare，请在 `.env` 中补充：

```bash
TUSHARE_TOKEN=your_token
TUSHARE_REQUEST_INTERVAL_SECONDS=1.5
```

## 登录保护

服务器远程访问时必须开启登录保护，本地开发可保持关闭。

生成密码哈希和随机会话密钥：

```bash
python scripts/generate_auth_secrets.py
```

脚本不会保存明文密码，会输出以下服务器环境变量：

```text
AUTH_ENABLED=true
AUTH_ADMIN_USERNAME=admin
AUTH_ADMIN_PASSWORD_HASH='pbkdf2_sha256$...'
AUTH_SESSION_SECRET='...'
AUTH_SESSION_TTL_HOURS=12
AUTH_COOKIE_SECURE=true
```

注意：

- `AUTH_ADMIN_PASSWORD_HASH` 是 PBKDF2-SHA256 哈希，不是明文密码。
- `AUTH_SESSION_SECRET` 至少 32 位，泄露或更换后现有会话会全部失效。
- `AUTH_COOKIE_SECURE=true` 只应配合 HTTPS 使用；本机 HTTP 开发环境使用 `false`。
- 登录 Cookie 使用 HttpOnly 和 SameSite=Lax，前端 JavaScript 无法直接读取。
- 登录连续失败 5 次后，同一来源在 15 分钟内会被临时限制。

## 生产服务器部署

生产环境使用独立的 `compose.production.yml`，不要直接使用本地 `docker-compose.yml`。

前置条件：

- 一台安装了 Docker Engine 和 Docker Compose 的 Linux 服务器。
- 一个已解析到服务器公网 IP 的域名。
- 云安全组或防火墙开放 TCP 22、80、443 和 UDP 443。
- 不要对公网开放 5432、6379、8000、5173。

首次部署步骤：

```bash
git clone https://github.com/tang003/ETFSystematicPortfolioPlatform.git
cd ETFSystematicPortfolioPlatform
cp .env.production.example .env.production
python3 scripts/generate_auth_secrets.py
```

编辑 `.env.production`：

- 把 `DOMAIN` 改为实际域名。
- 把 `ACME_EMAIL` 改为证书通知邮箱。
- 把 `POSTGRES_PASSWORD` 改为独立随机强密码。
- 粘贴认证密钥生成脚本输出的 `AUTH_*` 项。
- 填入服务器使用的 `TUSHARE_TOKEN`。

校验并启动：

```bash
docker compose --env-file .env.production -f compose.production.yml config -q
docker compose --env-file .env.production -f compose.production.yml up -d --build
docker compose --env-file .env.production -f compose.production.yml ps
```

说明：API 容器启动时会先执行 `alembic upgrade head`，迁移成功后才启动 Web API。如果迁移失败，API 会停止启动，需要先查看日志并修复数据库或迁移脚本。

查看日志：

```bash
docker compose --env-file .env.production -f compose.production.yml logs -f --tail=200
```

更新版本：

```bash
python3 scripts/backup_database.py --compose-file compose.production.yml --env-file .env.production
git pull --ff-only
docker compose --env-file .env.production -f compose.production.yml up -d --build
```

建议每次更新前先备份数据库。备份文件默认写入 `backups/`，并生成同名 `.sha256` 校验文件。

生产网络规则：

- Caddy 是唯一公网入口，自动申请和续期 TLS 证书，并把 HTTP 重定向到 HTTPS。
- PostgreSQL、Redis、FastAPI、workflow worker 和前端 Nginx 均不设置宿主机端口映射。
- 生产 API 关闭 Swagger、ReDoc 和 OpenAPI JSON。
- API 使用域名白名单拒绝异常 Host 请求，并只在生产反向代理链路中信任转发来源地址。
- 生产配置缺失鉴权、HTTPS Cookie、强数据库密码或域名白名单时，API 会拒绝启动。
- 生产全流程任务由独立 `worker` 服务执行，API 只负责创建 pending 任务和查询结果。
- API 的 `/health/ready` 会检查数据库和 worker 心跳；worker 失联时前端和 Caddy 不会继续把系统视为完全就绪。

说明：

- `TUSHARE_REQUEST_INTERVAL_SECONDS` 建议保持 `1.5` 或更高，适合共享账号节流。
- 当前容器启动时会把 `TUSHARE_*` 环境变量传递给 `api` 服务。
- 当前容器启动时会把 `AUTH_*` 环境变量传递给 `api` 服务。
- 当前容器启动时会把 `WORKFLOW_*` 环境变量传递给 `api` 和 `worker` 服务。
- 如果主要通过前端使用共享账号，建议同时开启“增量同步”，避免每次重复拉整段历史。

查看 worker 日志：

```bash
docker compose --env-file .env.production -f compose.production.yml logs -f --tail=200 worker
```

如果任务一直停留在 pending：

- 先检查 `docker compose --env-file .env.production -f compose.production.yml ps worker`。
- 再查看 worker 日志是否有 Tushare 限流、数据库连接或数据门禁错误。
- 如果 worker 曾异常退出，重新启动后会把未完成的 running 任务恢复为 pending 并重试当前步骤。

## 数据库备份与恢复

创建生产数据库备份：

```bash
python3 scripts/backup_database.py --compose-file compose.production.yml --env-file .env.production
```

脚本会执行以下动作：

- 使用 PostgreSQL 自定义格式生成 `.dump` 备份。
- 生成 `.dump.sha256` 校验文件。
- 使用 `pg_restore --list` 验证备份文件可读。
- 默认清理 30 天前的旧备份。

恢复数据库前必须确认目标数据库名。恢复会暂停 API 服务，执行 `pg_restore --clean --if-exists`，结束后重新启动 API：

```bash
python3 scripts/restore_database.py backups/quant_etf_YYYYMMDD_HHMMSS.dump \
  --compose-file compose.production.yml \
  --env-file .env.production \
  --confirm-database quant_etf
```

恢复保护规则：

- 备份文件必须存在。
- 同名 `.sha256` 文件必须存在，且校验值匹配。
- 备份必须能被 `pg_restore --list` 正常读取。
- `--confirm-database` 必须与正在运行的 PostgreSQL 容器内 `POSTGRES_DB` 完全一致。
- 恢复是破坏性操作，执行前必须再次确认最近一次备份可用。

服务端口：

- API：`8000`
- Web 控制台：`5173`
- PostgreSQL：宿主机 `5433` 映射到容器 `5432`
- Redis：宿主机 `6380` 映射到容器 `6379`

## 数据库初始化

首次启动 PostgreSQL 容器时会自动执行：

```text
sql/init_schema.sql
sql/seed_assets.sql
```

如果数据库 volume 已存在，初始化 SQL 不会重复执行。需要重新初始化时：

```bash
docker compose down -v
docker compose up --build
```

注意：`docker compose down -v` 会删除本地数据库 volume，只适合开发环境重建。生产服务器禁止在没有可用备份的情况下执行。

## 本地后端启动

如果不用 Docker 运行 API：

```bash
cd backend
python -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
uvicorn app.main:app --reload
```

需要确保 `.env` 中 PostgreSQL 地址可访问。

## 本地前端启动

```bash
cd frontend
npm install
npm run dev
```

访问：

```text
http://localhost:5173
```

Vite 开发服务器会把 `/api` 和 `/health` 代理到 `http://localhost:8000`。

前端控制台主要操作入口：

- 组合工作台：集中查看目标组合、当前持仓、偏离解释、定投建议和调仓建议。
- 全流程：创建后台任务，异步执行同步数据、计算因子、运行策略、风控调仓和生成报告；支持查看历史任务、取消任务和重试失败步骤。
- 严格数据门禁：全流程默认要求行情同步完整、数据更新到最近交易日且历史样本达到最低数量，否则不生成新的策略和调仓建议。
- 数据源控制：在“数据健康”和“全流程”页面中切换 `Tushare / AKShare / Eastmoney / Weekday fallback`，并控制请求间隔。
- ETF 池：查看当前 ETF 主数据，并支持通过前端批量导入扩展 ETF 池。
- ETF 池控制：支持按关键词、启用状态、资产类别、区域筛选，并支持逐只启用或停用研究对象。
- 数据健康：同步交易日历、同步 ETF 行情、检查缺失数据。
- 因子排名：按日期范围计算 Alpha 因子。
- 因子研究：计算 IC、Rank IC、因子相关性和分组未来收益。
- 目标组合：运行 `core_etf_rotation` 策略并生成目标持仓。
- 当前持仓：录入已有 ETF 持仓，系统按市值计算当前权重，并与目标组合对比生成加仓、减仓、持有或退出建议。
- 定投计划：设置定投期数和每期金额，按目标组合与当前持仓缺口生成本期投入建议。
- 风控调仓：按 `run_id` 执行风控检查并生成调仓单。
- 回测：运行等权买入持有回测或策略月度调仓回测，并查看净值曲线、指标和交易记录。
- 报告：生成并查看月度 Markdown 报告。

## 常见问题

### /health 返回 500

优先检查：

- PostgreSQL 容器是否启动。
- `.env` 中 `POSTGRES_HOST` 是否正确。
- 数据库名、用户名、密码是否与 `docker-compose.yml` 一致。

### 修改 SQL 后没有生效

PostgreSQL 初始化脚本只在空数据目录首次执行。开发阶段可以删除 volume 后重建。

### Swagger 打不开

确认 API 容器日志，或本地运行：

```bash
docker compose logs -f api
```

### 行情同步失败

当前行情同步优先使用 AKShare，并在 AKShare 失败时 fallback 到东方财富 K 线 HTTP 接口。若两个来源都不可用，`POST /api/market/sync` 会返回单个 ETF 的失败信息，并写入 `data_quality_log`。可以稍后重试，或缩小到单只 ETF 和较短日期区间验证。

如使用 Tushare：

- 先优先验证 `POST /api/calendar/sync` 的 `source=tushare`，因为交易日历更省权限、更省请求量。
- 再用 `POST /api/market/sync` 同步单只 ETF、较短日期范围。
- 如果返回权限不足，请检查当前积分权限是否覆盖 `fund_daily`。
- 如果是共享 Tushare 账号，优先把前端“请求间隔”设置为 `1.5` 秒或更高，并限制单次同步 ETF 数量。
- 如果只是日常补最新数据，优先开启“增量同步”，让系统只补最新缺口。

### 缺失数据检查不生效

缺失交易日检查依赖 `trading_calendar`。请先调用：

```bash
POST /api/calendar/sync
```

同步对应日期范围的交易日历后，再执行 `POST /api/data-quality/check`。

### 前端页面没有显示最新内容

如果刚修改了前端代码，但浏览器仍显示旧页面，或控制台提示某个导出不存在，通常是 Vite 开发服务缓存了旧模块。可以重启前端：

```bash
cd frontend
npm run dev
```

然后刷新浏览器页面。

### ETF 池为什么只有十几只

当前内置 SQL 默认提供的是一个“可跑通系统流程的起步样本池”，不是全市场 ETF 全量清单。

现在已经支持两种扩池方式：

- 修改 `sql/seed_assets.sql` 后重建空数据库
- 在前端“ETF 池”页面通过批量导入直接写入主数据

第二种方式更适合后续持续维护，不需要清空数据库。

### 持仓分析没有结果

优先检查：

- 是否已经生成目标组合，即 `/api/portfolio/target` 有数据。
- 前端“当前持仓”页的运行 ID 是否对应已有 `strategy_run.id`。
- 是否已经保存当前持仓快照。
- 如果没有持仓快照，调仓单会按当前权重为 0 处理。

### 定投计划没有建议

优先检查：

- 是否已经生成目标组合。
- 定投计划绑定的运行 ID 是否存在。
- 是否已经保存当前持仓快照。
- 如果没有当前持仓，系统会把当前权重按 0 处理，并按目标组合分配本期金额。
