# 10 系统流程说明

本文档从业务流程角度说明 ETF Systematic Portfolio Platform 的整体运行方式，方便后续自己维护，或把项目交给 AI 快速理解。

## 1. 平台目标

本系统面向个人中长期 ETF 投研与系统化资产配置，核心目标是把“数据获取、因子计算、策略生成、风控检查、调仓建议、回测复盘、报告输出”串成可追踪、可复用的流程。

系统当前不连接真实券商交易账户，不自动下单。所有调仓单都是建议单，需要人工确认。

服务器开启登录保护后，用户必须先通过管理员账号登录。健康检查和登录状态接口不读取投资数据，其余 ETF、持仓、定投、回测、报告和工作流接口全部受同一会话保护。

## 2. 总体流程

```text
同步 ETF 全市场基础池
→ 启用研究 ETF 池
→ 同步交易日历
→ 同步 ETF 行情
→ 清洗行情数据
→ 数据质量检查
→ ETF 对比和可交易性评分
→ 计算 Alpha 因子
→ 运行策略
→ 生成目标组合
→ 录入或读取当前持仓
→ 当前持仓分析
→ 定投计划分析
→ 执行风控检查
→ 生成调仓建议单
→ 回测与复盘
→ 生成月度报告
```

前端“全流程”页面会创建后台任务，由后端按步骤执行并持久化状态。

现在前端“数据健康”和“全流程”都可以直接控制数据源与节流参数，不需要再手动改请求体：

- `calendar_source`
- `market_source`
- `sync_scope`
- `request_interval_seconds`
- `incremental_sync`

ETF 池分为两层：

- 全市场基础池：保存系统已知的 ETF 代码、名称、交易所、分类、区域和风险等级。可在 ETF 池页面点击“同步全市场 ETF”从 AKShare/公开行情源导入，默认不进入策略研究。
- 启用研究池：`asset_master.enabled=true` 的 ETF，才会进入日常行情同步、因子计算、策略和回测。

行情同步不建议默认拉取市场全部 ETF。系统采用分层同步：

| 同步范围 | 说明 | 适用场景 |
| --- | --- | --- |
| `core` | 当前持仓、目标组合、定投建议涉及 ETF、启用 ETF 池去重合并 | 日常自动同步和人工同步默认选择 |
| `positions` | 最新持仓快照里的 ETF | 保证现价、盈亏和仓位分析可用 |
| `target` | 最新目标组合 ETF | 保证策略建议有行情基础 |
| `plans` | 定投建议中出现过的 ETF | 保证定投执行建议能更新 |
| `enabled` | ETF 池中启用的研究对象 | 扩大研究池时使用 |
| `all` | 资产主表中全部 ETF | 低频全量补数或初始化时使用 |

服务器自动任务建议每天收盘后调用 `scripts/run_core_market_sync.py --scope core --source tushare --max-symbols 30 --interval 1.5`。共享 Tushare token 时保留增量同步和请求间隔，避免短时间请求过多。

ETF 全市场基础池可以低频刷新，例如每周或每天盘后执行：

```bash
python3 scripts/run_etf_universe_sync.py --source akshare
```

## 6. 当前持仓录入规则

当前持仓模块面向手动录入券商持仓，不连接真实券商账户，也不操作资金。

用户必填：

- `symbol`：证券代码，例如 `513050`、`159928`。
- `quantity`：持仓数量/份额。
- `cost_price`：用户实际持仓成本价。

系统自动补全：

- `position_name`：从 `asset_master.name` 按代码查找。
- `asset_type`：从 `asset_master.asset_class` 推断为 `etf`、`stock`、`cash` 或 `other`。
- `current_price`：从 `market_data_clean` 中该代码最新交易日的 `close` 取得。
- `market_value`、`cost_basis`、`unrealized_pnl`、`weight`：由后端根据数量、现价、成本价和总市值统一计算。

当前持仓弹窗支持 `auto_sync=true` 的补全流程：

```text
用户输入 ETF 代码
→ 后端标准化代码，兼容 513050.SH / 159928.SZ
→ 如果 asset_master 不存在，自动登记为候选 ETF
→ 如果 market_data_clean 没有价格，自动同步最近 45 天行情
→ 返回名称、类型、现价和价格日期
```

因此当前持仓录入不要求 ETF 必须提前在精选池中；精选池用于研究和策略，持仓录入更偏向账户快照，应该优先保证用户能把真实持仓录进去。若外部行情源短时失败，前端允许临时填写现价，后续行情同步成功后再覆盖。

后续如果要支持债券、股票、现金等更复杂仓位，可以继续扩展 `asset_master.asset_class` 和持仓表字段，但基础录入仍应保持“用户只填必要信息”。

## 7. 全流程任务状态

后台任务由 `workflow_task.status` 表示总体状态：

| 状态 | 含义 |
| --- | --- |
| `pending` | 等待执行 |
| `running` | 执行中 |
| `success` | 全部步骤完成 |
| `partial_success` | 核心步骤完成，但有步骤被跳过，例如未生成报告 |
| `failed` | 某一步失败 |
| `cancelled` | 用户取消 |

每个步骤由 `workflow_task_step.status` 表示：

| 状态 | 含义 |
| --- | --- |
| `pending` | 未开始 |
| `running` | 当前正在执行 |
| `success` | 执行成功 |
| `failed` | 执行失败 |
| `skipped` | 被跳过 |
| `cancelled` | 取消时正在执行 |

取消任务当前是在步骤之间生效。如果某个行情同步请求正在等待外部接口返回，需要等该步骤结束后才会完全停止。

### 数据可信度门禁

全流程默认开启 `strict_data_validation`，在运行策略前执行以下检查：

1. 按用户填写代码或 ETF 池启用状态固定本次研究范围，并应用 `max_symbols` 限制。
2. 行情同步、质量检查和因子计算始终使用同一批 ETF，避免同步范围和计算范围不一致。
3. 行情同步全部失败时，任务立即失败；严格模式下任一 ETF 同步失败也会停止。
4. 读取交易日历中的最近开市日，确认每只 ETF 的 clean 行情已更新到该日期。
5. 确认每只 ETF 在所选日期范围内至少有 `minimum_history_bars` 条数据，默认 200 条。
6. 因子计算全部失败时停止；严格模式下不允许带着部分因子失败继续生成组合。

门禁失败时，步骤状态为 `failed`，具体代码、最新行情日期、样本数和失败原因保存在步骤结果中。补齐数据后可在前端点击“重试失败步骤”，系统不会重新执行此前已经成功的步骤。

## 8. 如何修改策略

当前核心策略代码在：

```text
backend/app/services/strategy_service.py
```

重点关注：

- `run_strategy`：策略运行入口。
- `load_factor_ranking`：读取某日因子排名。
- `construct_target_weights`：把排名转换成目标权重。
- `build_target_portfolio`：写入目标组合。

如果要新增策略，建议：

1. 在 `strategy_config` 中新增策略配置。
2. 在 `strategy_service.py` 中按 `strategy_code` 分发不同策略逻辑。
3. 保持返回结构包含 `run_id`, `signal_count`, `target_count`, `status`。
4. 更新 `docs/05_策略设计.md` 和 `docs/09_开发日志.md`。

## 9. 当前持仓如何影响调仓

当前持仓入口在前端“当前持仓”页面，对应后端服务：

```text
backend/app/services/holding_service.py
```

核心流程：

```text
用户录入持仓市值
→ 写入 portfolio_position
→ 系统计算 current_weight
→ 读取 target_portfolio.final_target_weight
→ 写入 holding_analysis_result
→ 生成 ADD / REDUCE / HOLD / REDUCE_OR_EXIT 建议
```

持仓分析只做建议，不操作真实资金。`action_suggestion` 的含义如下：

| 建议 | 中文含义 | 触发逻辑 |
| --- | --- | --- |
| `ADD` | 加仓 | 目标权重 - 当前权重 ≥ 3% |
| `REDUCE` | 减仓 | 当前权重 - 目标权重 ≥ 3% |
| `HOLD` | 持有 | 当前权重接近目标权重 |
| `REDUCE_OR_EXIT` | 减仓或退出 | 当前有持仓，但目标组合权重为 0 |

调仓建议单生成逻辑在：

```text
backend/app/services/risk_service.py
```

如果数据库已有当前持仓快照，`generate_rebalance_orders` 会优先使用真实 `current_weight`；如果没有持仓快照，则当前权重按 0 处理。用户输入的 `portfolio_value` 可以理解为本次组合总市值或计划管理的资金规模，用于把权重差额换算为建议买入或卖出金额。

## 10. 定投计划如何生成建议

定投计划入口在前端“定投计划”页面，对应后端服务：

```text
backend/app/services/investment_plan_service.py
```

核心流程：

```text
用户设置计划名称、开始日期、期数、准备资金、目标年化和执行模式
→ 写入 investment_plan
→ 系统按 准备资金 / 期数 计算每期金额
→ 读取目标组合 target_portfolio
→ 读取当前持仓 portfolio_position
→ 计算每只 ETF 的权重缺口 target_weight - current_weight
→ 优先把本期金额分配给低配 ETF
→ 写入 investment_plan_suggestion
```

当前定投算法是可解释的基础版：

- 如果某些 ETF 的目标权重大于当前权重，则只在这些低配 ETF 之间分配本期金额。
- 分配比例按权重缺口大小决定。
- 如果没有低配 ETF，则按目标组合权重分配本期金额。
- `target_annual_return` 是策略目标参数，不代表收益承诺。
- `investment_mode` 用于区分固定节奏、回撤增强、信号驱动和未来自动执行预览。
- 定投建议只输出金额建议，不扣款，不下单，不连接真实资金账户。
- 后续如果开放自动买卖接口，应单独增加执行层：交易权限、资金校验、风控开关、撤单/失败处理、审计日志和人工确认。

示例：

```text
当前持仓：
510300 当前 60%，目标 40%
511880 当前 40%，目标 35%
511010 当前 0%，目标 25%

准备资金 10000 元，投入 12 期：
系统计算每期约 833.33 元。

511010 是唯一低配 ETF，本期建议全部投入 511010。
```

## 11. 如何接入真实行情 API

当前行情服务在：

```text
backend/app/services/market_service.py
```

当前流程：

```text
AKShare
→ 失败时 fallback 到东方财富 K 线接口
→ market_data_raw
→ market_data_clean
```

当前已经预留并接入以下数据源选择：

```text
source=akshare
source=eastmoney
source=tushare
```

其中：

- `akshare` 适合快速开发。
- `eastmoney` 适合作为 AKShare 的备用源。
- `tushare` 适合后续逐步切换到更稳定、可控的真实数据源。

当前 Tushare 实现位置：

```text
backend/app/services/tushare_service.py
```

后续如果继续扩展 Tushare、聚宽、Wind 或其他真实数据源，建议继续保持独立数据源函数，例如：

```text
fetch_tushare_daily_bars
fetch_xxx_daily_bars
```

然后在 `fetch_etf_daily_bars` 中按优先级选择数据源。无论外部 API 返回什么格式，最终都要归一化为：

```text
symbol
trade_date
open
high
low
close
volume
amount
source
raw_payload
```

如果使用共享 Tushare 账号，建议的操作顺序是：

1. 先同步交易日历，验证 token 可用。
2. 再同步单只 ETF、较短日期范围的行情。
3. 默认开启增量同步，只补最新缺口。
4. 确认权限和节流策略后，再考虑批量同步。

## 13. 策略调仓回测流程

回测入口在前端“回测”页面，对应后端服务：

```text
backend/app/services/backtest_service.py
```

当前支持两种模式：

| 模式 | 含义 |
| --- | --- |
| `equal_weight_buy_and_hold` | 首日等权买入并持有，用作基线 |
| `core_etf_rotation_monthly` | 每月按策略目标权重调仓，支持每月追加资金 |

策略月度调仓流程：

```text
读取回测区间行情
→ 读取 factor_daily
→ 每月第一个可交易日读取最近 Alpha 排名
→ 生成目标权重
→ 卖出高配资产
→ 买入低配资产
→ 月份切换时加入 monthly_contribution
→ 记录交易和每日净值
→ 输出累计收益、年化收益、最大回撤、波动率、夏普比率、交易次数
```

当前回测仍是分析工具，不代表未来收益，也不会触发真实交易。

## 14. 常见排查路径

### 前端页面显示异常

优先检查：

- `frontend/src/styles.css` 是否缺少对应栅格类。
- 页面是否在过窄宽度下仍强制左右并排。
- 表格列宽是否过大或过小。
- `npm run build` 是否通过。

### 全流程任务失败

优先查看：

- 前端“全流程”页面的任务日志。
- `workflow_task.error_message`。
- `workflow_task_step.message`。
- `docker compose logs -f api`。

### 行情同步失败

行情同步依赖外部接口。AKShare 或东方财富短时不可用时，任务会记录失败原因。可以稍后重试失败步骤，或缩小日期范围和 ETF 数量。

### 报告为空或内容不完整

优先确认：

- 是否已经有 `strategy_run`。
- 指定的 `run_id` 是否存在。
- 对应 `run_id` 是否有目标组合、风控结果和调仓单。
- 回测摘要为空不影响报告生成。

### 定投计划没有建议

优先确认：

- 是否已经有目标组合。
- 定投计划绑定的 `run_id` 是否存在。
- 是否已经录入当前持仓快照。
- 如果没有当前持仓，系统会按当前权重为 0 处理。

## 15. 后续扩展路线

优先级建议：

1. 任务队列升级：从 FastAPI `BackgroundTasks` 升级为 Redis/Celery 或 APScheduler 持久队列。
2. 数据源增强：接入 Tushare 或其他稳定行情 API。
3. 策略增强：支持多策略、多参数、多组合版本。
4. 报告增强：增加收益归因、风险暴露和历史对比。
5. 权限与账户：未来如需多人使用，再增加登录、权限和审计。
