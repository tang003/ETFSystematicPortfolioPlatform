# 10 系统流程说明

本文从业务流程角度说明平台如何运转，便于自己维护或交给 AI 快速接手。

## 1. 总体流程

```text
ETF 基础池
 -> 启用研究 ETF
 -> 同步交易日历
 -> 同步 ETF 行情
 -> 清洗和质量检查
 -> 计算因子
 -> ETF 对比和可交易性评分
 -> 运行策略
 -> 生成目标组合
 -> 录入当前持仓
 -> 持仓分析
 -> 定投计划
 -> 风控检查
 -> 调仓建议
 -> 回测
 -> 报告
 -> AI 投研总结
```

## 2. ETF 池

ETF 池分两层：

- 全市场基础池：保存系统知道的 ETF 代码、名称、分类、交易所、风险等级。
- 启用研究池：`asset_master.enabled=true` 的 ETF，才进入日常行情、因子、策略和回测。

这样设计是为了避免每次都同步全市场 ETF，尤其在共享 Tushare token 时减少请求压力。

ETF 基础池同步说明：
- `source=auto` 的顺序是东方财富分页源、Tushare `fund_basic`、AKShare。
- 基础池同步结果写入 `asset_master`，只保存 ETF 代码、名称、交易所、分类、基金公司、跟踪标的、费率等档案信息。
- 基础池同步不保存历史行情；历史行情仍在“数据健康”“ETF 详情”或“全流程”中按代码和日期范围同步。

## 3. 行情同步

同步范围：

| 范围 | 含义 |
| --- | --- |
| `core` | 当前持仓、目标组合、定投建议、启用研究池的合集 |
| `positions` | 最新持仓里的 ETF |
| `target` | 最新目标组合里的 ETF |
| `plans` | 定投建议涉及的 ETF |
| `enabled` | 启用研究池 |
| `all` | 资产主表全部 ETF |

日常建议使用 `core`。

## 4. 当前持仓录入

当前推荐手动录入，因为东方财富、同花顺等券商/客户端的持仓同步涉及登录、隐私、接口稳定性和合规问题。

用户主要填写：

- ETF 代码。
- 持仓数量。
- 成本价。

系统补全：

- 名称。
- 类型。
- 现价。
- 市值。
- 权重。
- 成本金额。
- 浮盈亏。

如果代码不在 ETF 池，系统会自动登记候选 ETF，并尝试同步最近行情。

## 5. 持仓分析

持仓分析需要目标组合作为参照。

输出：

- 当前权重。
- 目标权重。
- 权重差。
- 建议动作：加仓、减仓、持有、减仓或退出。
- 原因。

如果还没有目标组合，需要先运行策略或全流程。

## 6. 定投计划

用户可以设置：

- 准备投入总资金。
- 投入周期，例如 12 个月。
- 每月投入金额。
- 目标年化收益。
- 执行模式。

系统会根据目标组合和当前持仓缺口生成每期建议。

当前不会自动扣款或自动交易。

## 7. AI 投研委员会

AI 投研页面流程：

```text
输入 ETF 代码
 -> 自动补全数据
 -> 读取本地 ETF 详情、因子、持仓、目标组合、持仓分析
 -> 市场环境 Agent
 -> 技术趋势 Agent
 -> 流动性 Agent
 -> 组合持仓 Agent
 -> 风险控制 Agent
 -> 复合经理 Agent
 -> DeepSeek 中文总结
 -> 保存历史记录
```

注意：

- DeepSeek 只根据系统提供的数据总结。
- 如果 DeepSeek 不可用，仍返回规则型结论。
- 每次分析保存到 `agent_analysis_log`。
- 当前不使用 AI 自动下单。

## 8. 全流程页面

全流程用于把多个步骤串起来：

1. 同步交易日历。
2. 同步行情。
3. 检查数据质量。
4. 计算因子。
5. 运行策略。
6. 生成目标组合。
7. 风控检查。
8. 生成调仓建议。
9. 可选生成报告。

生产环境由 worker 执行任务，前端轮询任务状态。

## 9. 报告

报告是 Markdown 格式，中文为主，保留必要英文术语，例如：

- Alpha
- Target Portfolio
- Rebalance Orders
- Backtest
- Drawdown

报告用于复盘，不是投资承诺。

## 10. 后续接真实交易的原则

未来若接入券商或交易接口，必须分层设计：

- 只读持仓同步。
- 交易前人工确认。
- 风控强制检查。
- 订单审计日志。
- 撤单和失败重试。
- 异常暂停开关。

不能让 AI 直接绕过风控下单。
