# 10 系统流程说明

本文档从业务流程角度说明 ETF Systematic Portfolio Platform 的整体运行方式，方便后续自己维护，或把项目交给 AI 快速理解。

## 1. 平台目标

本系统面向个人中长期 ETF 投研与系统化资产配置，核心目标是把“数据获取、因子计算、策略生成、风控检查、调仓建议、回测复盘、报告输出”串成可追踪、可复用的流程。

系统当前不连接真实券商交易账户，不自动下单。所有调仓单都是建议单，需要人工确认。

## 2. 总体流程

```text
ETF 池
→ 同步交易日历
→ 同步 ETF 行情
→ 清洗行情数据
→ 数据质量检查
→ 计算 Alpha 因子
→ 运行策略
→ 生成目标组合
→ 执行风控检查
→ 生成调仓建议单
→ 回测与复盘
→ 生成月度报告
```

前端“全流程”页面会创建后台任务，由后端按步骤执行并持久化状态。

## 3. 前端页面与业务关系

| 页面 | 作用 | 主要 API |
| --- | --- | --- |
| 总览 | 查看核心指标、Alpha 排名、目标权重和调仓单 | `/api/assets`, `/api/factors/ranking`, `/api/portfolio/target` |
| 全流程 | 创建后台任务、查看任务历史、取消任务、重试失败步骤 | `/api/workflows/*` |
| ETF 池 | 查看当前启用 ETF 主数据 | `/api/assets` |
| 数据健康 | 同步交易日历、同步行情、检查缺失数据 | `/api/calendar/sync`, `/api/market/sync`, `/api/data-quality/check` |
| 因子排名 | 计算因子并查看 Alpha 排名 | `/api/factors/calculate`, `/api/factors/ranking` |
| 目标组合 | 运行策略并查看目标组合 | `/api/strategies/run`, `/api/portfolio/target` |
| 风控调仓 | 执行风控检查并生成调仓单 | `/api/risk/check`, `/api/rebalance/generate` |
| 回测 | 运行回测并查看净值曲线和指标 | `/api/backtest/run`, `/api/backtest/*` |
| 报告 | 生成和查看月度报告 | `/api/reports/monthly`, `/api/reports` |

## 4. 后端模块职责

| 目录 | 职责 |
| --- | --- |
| `backend/app/api` | FastAPI 路由层，只负责接收请求和返回响应 |
| `backend/app/services` | 核心业务逻辑层，策略、风控、回测、报告都在这里 |
| `backend/app/models` | SQLAlchemy ORM，对应数据库表 |
| `backend/app/schemas` | Pydantic 请求和响应模型 |
| `backend/app/core` | 配置、数据库连接、日志、调度器等基础能力 |

## 6. 全流程任务状态

后台任务由 `workflow_task.status` 表示总体状态：

| 状态 | 含义 |
| --- | --- |
| `pending` | 等待执行 |
| `running` | 执行中 |
| `success` | 全部步骤完成 |
| `partial_success` | 核心步骤完成，但有步骤被跳过，例如未生成报告 |
| `failed` | 某一步失败 |
| `cancelled` | 用户取消 |

每个步骤由 `workflow_task_step.status` 表示：

| 状态 | 含义 |
| --- | --- |
| `pending` | 未开始 |
| `running` | 当前正在执行 |
| `success` | 执行成功 |
| `failed` | 执行失败 |
| `skipped` | 被跳过 |
| `cancelled` | 取消时正在执行 |

取消任务当前是在步骤之间生效。如果某个行情同步请求正在等待外部接口返回，需要等该步骤结束后才会完全停止。

## 7. 如何修改策略

当前核心策略代码在：

```text
backend/app/services/strategy_service.py
```

重点关注：

- `run_strategy`：策略运行入口。
- `load_factor_ranking`：读取某日因子排名。
- `construct_target_weights`：把排名转换成目标权重。
- `build_target_portfolio`：写入目标组合。

如果要新增策略，建议：

1. 在 `strategy_config` 中新增策略配置。
2. 在 `strategy_service.py` 中按 `strategy_code` 分发不同策略逻辑。
3. 保持返回结构包含 `run_id`, `signal_count`, `target_count`, `status`。
4. 更新 `docs/05_策略设计.md` 和 `docs/09_开发日志.md`。

## 8. 如何接入真实行情 API

当前行情服务在：

```text
backend/app/services/market_service.py
```

当前流程：

```text
AKShare
→ 失败时 fallback 到东方财富 K 线接口
→ market_data_raw
→ market_data_clean
```

未来接入 Tushare、聚宽、Wind 或其他真实数据源时，建议新增独立数据源函数，例如：

```text
fetch_tushare_daily_bars
fetch_xxx_daily_bars
```

然后在 `fetch_etf_daily_bars` 中按优先级选择数据源。无论外部 API 返回什么格式，最终都要归一化为：

```text
symbol
trade_date
open
high
low
close
volume
amount
source
raw_payload
```

## 9. 报告生成流程

报告服务在：

```text
backend/app/services/report_service.py
```

报告数据来源：

```text
strategy_run
alpha_signal
target_portfolio
risk_check_result
rebalance_order
backtest_run
backtest_metrics
```

报告以 Markdown 保存到 `report_log.content_markdown`，前端报告页负责渲染 Markdown 内容。

报告文字以中文为主，保留必要英文术语，例如：

- Alpha Ranking
- Target Portfolio
- Risk Check
- Rebalance Orders
- Backtest Snapshot
- Monthly Rebalance Report

## 10. 常见排查路径

### 前端页面显示异常

优先检查：

- `frontend/src/styles.css` 是否缺少对应栅格类。
- 页面是否在过窄宽度下仍强制左右并排。
- 表格列宽是否过大或过小。
- `npm run build` 是否通过。

### 全流程任务失败

优先查看：

- 前端“全流程”页面的任务日志。
- `workflow_task.error_message`。
- `workflow_task_step.message`。
- `docker compose logs -f api`。

### 行情同步失败

行情同步依赖外部接口。AKShare 或东方财富短时不可用时，任务会记录失败原因。可以稍后重试失败步骤，或缩小日期范围和 ETF 数量。

### 报告为空或内容不完整

优先确认：

- 是否已经有 `strategy_run`。
- 指定的 `run_id` 是否存在。
- 对应 `run_id` 是否有目标组合、风控结果和调仓单。
- 回测摘要为空不影响报告生成。

## 11. 后续扩展路线

优先级建议：

1. 任务队列升级：从 FastAPI `BackgroundTasks` 升级为 Redis/Celery 或 APScheduler 持久队列。
2. 数据源增强：接入 Tushare 或其他稳定行情 API。
3. 策略增强：支持多策略、多参数、多组合版本。
4. 报告增强：增加收益归因、风险暴露和历史对比。
5. 权限与账户：未来如需多人使用，再增加登录、权限和审计。
