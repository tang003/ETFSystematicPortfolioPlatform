# 10 系统流程说明

本文从业务流程角度说明平台如何运转，便于自己维护或交给 AI 快速接手。

## 1. 总体流程

```text
ETF 基础池
 -> 启用研究 ETF
 -> 同步交易日历
 -> 同步 ETF 行情
 -> 清洗和质量检查
 -> 计算因子
 -> ETF 对比和可交易性评分
 -> 运行策略
 -> 生成目标组合
 -> 录入当前持仓
 -> 持仓分析
 -> 定投计划
 -> 风控检查
 -> 调仓建议
 -> 回测
 -> 报告
 -> AI 投研总结
```

## 2. ETF 池

ETF 池分两层：

- 全市场基础池：保存系统知道的 ETF 代码、名称、分类、交易所、风险等级。
- 启用研究池：`asset_master.enabled=true` 的 ETF，才进入日常行情、因子、策略和回测。

这样设计是为了避免每次都同步全市场 ETF，尤其在共享 Tushare token 时减少请求压力。

ETF 基础池同步说明：
- 当前正式同步源为 Tushare `fund_basic`；`source=auto` 在正式口径下等同优先 Tushare。
- AKShare / 东方财富旧代码只作为历史遗留保留，业务入口已停用。
- 基础池同步结果写入 `asset_master`，只保存 ETF 代码、名称、交易所、分类、基金公司、跟踪标的、费率等档案信息。
- 基础池同步不保存历史行情；历史行情仍在“数据健康”“ETF 详情”或“全流程”中按代码和日期范围同步。

## 3. 行情同步

系统中统一使用“分析周期”表示历史回看窗口。

分析周期不是未来持有期，也不是定投计划周期。详细口径见 `docs/15_分析周期口径说明.md`。

同步范围：

| 范围 | 含义 |
| --- | --- |
| `core` | 当前持仓、目标组合、定投建议、启用研究池的合集 |
| `positions` | 最新持仓里的 ETF |
| `target` | 最新目标组合里的 ETF |
| `plans` | 定投建议涉及的 ETF |
| `enabled` | 启用研究池 |
| `all` | 资产主表全部 ETF |

日常建议使用 `core`。

数据健康页会按当前分析周期展示每只 ETF 的行情覆盖情况：

- 最新行情日。
- 当前分析区间内已有样本数。
- 交易日历预期样本数。
- 覆盖率。
- 策略可用状态：可用、样本不足、待补行情。

如果 ETF 状态为“样本不足”或“待补行情”，应先同步行情并重新检查，再进入因子、策略、回测和 AI 投研流程。

数据健康页会根据当前分析周期生成 `recommended_sync_symbols`，前端“补齐缺口”按钮只同步这些 ETF，避免为了少数缺失样本重复请求整个 ETF 池。共享 Tushare token 场景下，建议先保留较小的同步数量和请求间隔，分批把建议补齐列表处理完。

补齐缺口已接入后台任务：

1. 前端调用 `POST /api/workflows/market-repair` 创建任务。
2. 后台任务依次执行交易日历同步、ETF 行情补齐、补齐结果检查。
3. 前端按 `task_id` 轮询任务状态。
4. 任务完成后自动刷新数据健康状态、同步计划和质量日志。

这样可以避免浏览器长时间等待外部数据源请求，生产环境由 worker 负责执行。

## 4. 当前持仓录入

当前推荐手动录入，因为东方财富、同花顺等券商/客户端的持仓同步涉及登录、隐私、接口稳定性和合规问题。

用户主要填写：

- ETF 代码。
- 持仓数量。
- 成本价。

系统补全：

- 名称。
- 类型。
- 现价。
- 市值。
- 权重。
- 成本金额。
- 浮盈亏。

如果代码不在 ETF 池，系统会自动登记候选 ETF，并尝试同步最近行情。

## 5. 持仓分析

持仓分析需要目标组合作为参照。

输出：

- 当前权重。
- 目标权重。
- 权重差。
- 建议动作：加仓、减仓、持有、减仓或退出。
- 原因。

如果还没有目标组合，需要先运行策略或全流程。

## 6. 定投计划

用户可以设置：

- 准备投入总资金。
- 投入周期，例如 12 个月。
- 每月投入金额。
- 目标年化收益。
- 执行模式。

系统会根据目标组合和当前持仓缺口生成每期建议。

当前不会自动扣款或自动交易。

## 7. AI 投研委员会

AI 投研页面流程：

```text
输入 ETF 代码
 -> 自动补全数据
 -> 读取本地 ETF 详情、因子、持仓、目标组合、持仓分析
 -> 市场环境 Agent
 -> 技术趋势 Agent
 -> 流动性 Agent
 -> 组合持仓 Agent
 -> 风险控制 Agent
 -> 复合经理 Agent
 -> DeepSeek 中文总结
 -> 保存历史记录
```

注意：

- DeepSeek 只根据系统提供的数据总结。
- 如果 DeepSeek 不可用，仍返回规则型结论。
- 每次分析保存到 `agent_analysis_log`。
- 当前不使用 AI 自动下单。

## 8. 全流程页面

全流程用于把多个步骤串起来：

1. 同步交易日历。
2. 同步行情。
3. 检查数据质量。
4. 计算因子。
5. 运行策略。
6. 生成目标组合。
7. 风控检查。
8. 生成调仓建议。
9. 可选生成报告。

生产环境由 worker 执行任务，前端轮询任务状态。

系统状态页的“任务中心”用于统一观察后台任务：

- 区分全流程、历史初始化和行情补齐任务。
- 展示等待/执行、成功和异常数量。
- 展示每个任务的进度、当前步骤、耗时和处理对象。
- 展开任务可查看每个步骤的状态、耗时和结果摘要。
- 对 `daily_maintenance` 任务，展开后会把行情同步、因子计算、策略运行、风控检查、调仓建议和报告生成结果拆成中文摘要卡片。
- 常见失败原因会转成中文提示，便于判断是否需要补数据、调高请求间隔或重试。

全流程页面中的“分析周期”会同时影响交易日历、行情同步、质量检查、因子计算、策略运行和报告生成。默认建议使用最近 1 年；中期配置可使用最近 3 年；新发 ETF 过多时应注意样本不足。

## 9. 每日自动维护

生产环境可以开启 worker 内置的每日维护任务。它适合在收盘后自动刷新本地数据和研究结果，减少手动点击。

每日维护默认流程：

1. 按 `DAILY_MAINTENANCE_LOOKBACK_DAYS` 同步最近交易日历。
2. 按 `DAILY_MAINTENANCE_SCOPE` 增量同步 ETF 日线行情。
3. 重算启用研究 ETF 的因子。
4. 使用 `DAILY_MAINTENANCE_STRATEGY_CODE` 运行策略。
5. 对最新目标组合执行风控检查。
6. 按 `DAILY_MAINTENANCE_PORTFOLIO_VALUE` 生成调仓建议单。
7. 如果 `DAILY_MAINTENANCE_GENERATE_REPORT=true`，生成 Markdown 报告。

关键边界：

- 它只生成研究结果，不连接券商，不自动买卖。
- 调度只运行在 worker 进程中，避免 API 多实例重复执行。
- Redis 锁 `daily_maintenance:lock` 用于防止重复运行。
- 默认关闭，需要在 `.env.production` 设置 `DAILY_MAINTENANCE_ENABLED=true` 后重新部署或重启 worker。
- 系统状态页可以点击“立即执行”手动创建一次 `daily_maintenance` 后台任务，任务会进入任务中心，由 worker 执行。
- 系统状态页会展示最近一次维护摘要，包括处理 ETF 数量、是否新增行情、因子行数、策略 run_id、目标持仓数量、风控结果、调仓建议数量和报告 ID。

## 10. 报告

报告是 Markdown 格式，中文为主，保留必要英文术语，例如：

- Alpha
- Target Portfolio
- Rebalance Orders
- Backtest
- Drawdown

报告用于复盘，不是投资承诺。

## 11. 后续接真实交易的原则

未来若接入券商或交易接口，必须分层设计：

- 只读持仓同步。
- 交易前人工确认。
- 风控强制检查。
- 订单审计日志。
- 撤单和失败重试。
- 异常暂停开关。

不能让 AI 直接绕过风控下单。
