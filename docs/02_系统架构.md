# 02 系统架构

## 当前架构

```text
ETFSystematicPortfolioPlatform
├── backend
│   └── app
│       ├── api          # FastAPI 路由
│       ├── core         # 配置、数据库、日志、调度器
│       ├── models       # SQLAlchemy ORM
│       ├── schemas      # Pydantic 请求和响应模型
│       └── services     # 业务服务
├── frontend             # Vue 3 + Element Plus 控制台
├── sql                  # 首次初始化建表和种子数据
├── scripts              # 运维脚本，例如密钥生成、备份和恢复
├── docs                 # 设计文档、系统说明和开发日志
└── docker-compose.yml   # PostgreSQL、Redis、API、Web 编排
```

系统采用前后端分离结构。后端负责数据、策略、风控、回测和报告，前端负责操作入口和结果展示，PostgreSQL 保存核心业务数据，Redis 当前主要为后续任务队列扩展预留。

生产环境使用 `compose.production.yml` 独立编排。公网只暴露 Caddy 的 80/443 端口，Caddy 负责自动 HTTPS 和安全响应头；前端 Nginx、FastAPI、workflow worker、PostgreSQL、Redis 只位于 Docker 内部网络，不直接暴露到公网。API 和 worker 容器启动时会先执行 `alembic upgrade head`，确保数据库结构随代码版本前进。

```text
Internet
→ Caddy :80/:443
→ frontend Nginx :80（Docker 内网）
→ FastAPI :8000（Docker 内网）
→ PostgreSQL / Redis / workflow worker（Docker 内网）
```

## 后端分层

### backend/app/core

- `config.py`：统一读取 `.env` 环境变量，生成数据库连接串并管理鉴权配置。
- `security.py`：PBKDF2 密码哈希、HMAC 签名会话和安全校验。
- `database.py`：创建 SQLAlchemy engine、session 和 `get_db` 依赖。
- `logging.py`：初始化 loguru 日志。
- `scheduler.py`：预留 APScheduler 定时任务入口。

### backend/app/models

每张核心业务表对应一个 SQLAlchemy model。当前已覆盖资产、行情、数据质量、因子、策略、组合、持仓、风控、调仓、回测、报告和工作流任务。

### backend/app/schemas

Pydantic 请求和响应模型。API 入参、出参应优先在 schema 中定义，避免在路由层直接拼散乱 dict。

### backend/app/api

API 层只负责参数接收、依赖注入、调用 service 和返回响应，不承载复杂业务逻辑。

当前主要路由文件：

- `health_api.py`：健康检查。
- `auth_api.py`：单管理员登录、会话状态、退出和 API 鉴权依赖。
- `asset_api.py`：ETF 主数据查询。
- `calendar_api.py`：交易日历同步和查询。
- `market_api.py`：行情同步、raw/clean 行情查询。
- `data_quality_api.py`：数据质量检查、日志和状态查询。
- `factor_api.py`：因子计算、因子详情和 Alpha 排名。
- `strategy_api.py`：策略配置、策略运行和最新信号。
- `portfolio_api.py`：目标组合、当前持仓快照和持仓分析。
- `risk_api.py`：风控检查。
- `rebalance_api.py`：调仓建议单生成和查询。
- `backtest_api.py`：回测运行、净值曲线和指标查询。
- `report_api.py`：月度报告生成和查询。
- `workflow_api.py`：全流程后台任务、任务日志、取消和失败重试。

### backend/app/services

业务逻辑层。复杂逻辑应放入 service，方便测试、复用和后续替换实现。

当前重点服务：

- `market_service.py`：行情同步、fallback 数据源和清洗。
- `factor_service.py`：因子计算和 Alpha 排名。
- `factor_research_service.py`：IC、Rank IC、因子相关性和分组收益研究。
- `strategy_service.py`：策略运行和目标组合生成。
- `holding_service.py`：当前持仓快照、持仓分析、动作建议。
- `investment_plan_service.py`：定投计划、本期投入建议和缺口分配。
- `risk_service.py`：风控检查和调仓单生成。
- `backtest_service.py`：基础买入持有回测、策略月度调仓回测和绩效指标计算。
- `report_service.py`：月度 Markdown 报告。
- `workflow_service.py`：后台任务编排。

### backend/app/worker.py

生产环境的独立工作流执行进程。API 创建任务后只写入 `workflow_task` pending 记录，worker 轮询数据库认领任务并执行；worker 会把心跳写入 Redis，`/health/ready` 会检查该心跳，避免任务执行进程失联时仍被认为完全可用。本地开发默认 `WORKFLOW_EXECUTION_MODE=inline`，保持创建任务后立即由 API 后台执行。

### backend/alembic

数据库迁移目录。`20260710_0001_current_schema_baseline.py` 是当前 SQLAlchemy 模型的迁移基线，用于让后续新增字段、表和索引都通过 Alembic 版本化管理。生产容器启动会自动执行最新迁移。

## 前端结构

前端位于 `frontend/`：

```text
frontend/src
├── api       # axios 客户端和 API 类型
├── router    # Vue Router 页面路由
├── views     # 业务页面
├── App.vue   # 主布局和左侧导航
└── styles.css
```

当前页面包括登录、总览、组合工作台、全流程、ETF 池、数据健康、因子排名、因子研究、目标组合、当前持仓、定投计划、风控调仓、回测和报告。

服务器开启 `AUTH_ENABLED=true` 后，健康检查和认证接口保持公开，其余 `/api/*` 业务路由统一要求有效的 HttpOnly Cookie 会话。前端路由守卫会在进入业务页面前检查会话，401 响应会跳回登录页。

## 数据流

基础主数据流：

```text
seed_assets.sql
→ asset_master / strategy_config / risk_rule
→ SQLAlchemy service
→ FastAPI API
→ 前端页面或外部调用方
```

行情数据闭环：

```text
AKShare
→ POST /api/market/sync
→ AKShare 失败时 fallback 到东方财富 K 线 HTTP 接口
→ market_data_raw
→ market_data_clean
→ data_quality_log
→ GET /api/market/bars/{symbol}
```

交易日历流：

```text
AKShare 交易日历
→ POST /api/calendar/sync
→ trading_calendar
→ 数据质量检查判断 open day 缺失
```

当前完整业务数据流：

```text
外部行情源
→ market_data_raw
→ 数据清洗
→ market_data_clean
→ 数据质量日志
→ factor_daily
→ alpha_signal
→ target_portfolio
→ portfolio_position
→ holding_analysis_result
→ investment_plan / investment_plan_suggestion
→ risk_check_result
→ rebalance_order
→ backtest_run / backtest_metrics
→ report_log
```

## 策略流

```text
读取 strategy_config
→ 创建 strategy_run
→ 读取 factor_daily
→ 生成 alpha_signal
→ 生成 target_portfolio.raw_target_weight
→ 交给风控引擎修正
→ 保存 final_target_weight
```

## 持仓分析流

当前持仓用于把目标组合转成对用户已有仓位更有意义的建议：

```text
用户录入当前持仓
→ portfolio_position
→ 计算 current_weight
→ 对比 target_portfolio.final_target_weight
→ holding_analysis_result
→ 输出 ADD / REDUCE / HOLD / REDUCE_OR_EXIT
```

## 风控调仓流

风控独立于策略，调仓独立于真实交易：

```text
策略原始仓位
→ 风控规则 risk_rule
→ 风控检查 risk_check_result
→ 风控后目标仓位
→ 读取当前持仓权重
→ 生成调仓建议单 rebalance_order
```

## 回测流

回测模块输出：

- `backtest_run`
- `backtest_equity_curve`
- `backtest_trade`
- `backtest_metrics`

当前已实现基础买入持有回测和策略月度调仓回测。策略月度调仓回测会在每月第一个可交易日读取最近因子排名，生成目标权重，并按目标权重买卖。

## 报告流

报告模块生成 Markdown 内容并保存到：

- `report_log.content_markdown`
- 可选文件路径 `report_log.file_path`

报告内容包括策略运行摘要、Alpha 排名、目标组合、风控检查、调仓建议、回测摘要和系统备注。
