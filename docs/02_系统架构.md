# 02 系统架构

## 总览

本项目采用前后端分离架构：

```text
浏览器 Vue 前端
  -> FastAPI 后端
  -> PostgreSQL 数据库
  -> Redis 任务状态/心跳
  -> worker 后台任务
  -> 外部数据源 Tushare / DeepSeek / 聚合财经新闻
```

生产环境由外部 Caddy 负责 HTTPS 和域名反向代理，项目自身前端容器只绑定服务器本机端口，避免影响服务器上的其他项目。

## 技术栈

后端：

- Python 3.11
- FastAPI
- SQLAlchemy
- Pydantic
- Alembic
- PostgreSQL
- Redis
- requests / pandas
- akshare 仍在依赖和历史代码中保留，当前正式业务入口不使用

前端：

- Vue 3
- TypeScript
- Element Plus
- ECharts
- Vite
- Axios

部署：

- Docker Compose
- Nginx 前端容器
- 外部 Caddy 反向代理
- GitHub 拉取代码部署

## 后端目录

| 目录 | 职责 |
| --- | --- |
| `backend/app/api` | FastAPI 路由层，只处理请求、响应和 HTTP 异常 |
| `backend/app/services` | 业务逻辑层，策略、风控、回测、报告、AI 分析都在这里 |
| `backend/app/models` | SQLAlchemy ORM 模型 |
| `backend/app/schemas` | Pydantic 请求和响应模型 |
| `backend/app/core` | 配置、数据库、日志、安全、调度 |
| `backend/alembic` | 数据库迁移 |
| `backend/tests` | 后端测试 |

当前正式 ETF 数据源说明：

- ETF 基础池、主资料、交易日历、日线行情、基金净值、基金份额和指数行情以 Tushare 为正式口径。
- DeepSeek 只用于 AI 投研总结和表达增强。
- 聚合数据财经新闻用于低频新闻上下文。
- AKShare / Eastmoney 旧函数保留但入口停用，不应进入正式清洗行情。

## 任务执行模式

本地开发：

- 默认可以由 API inline 执行工作流。

生产环境：

- API 接收任务并入库。
- worker 轮询 pending 任务并执行。
- Redis 用于 worker 心跳和健康检查。

## 鉴权架构

生产环境开启：

- `AUTH_ENABLED=true`
- 管理员用户名密码。
- HttpOnly Cookie 会话。
- 登录限流。
- 除 `/health` 和 `/api/auth/*` 外，业务 API 都需要登录。

## 数据流

典型数据流：

```text
ETF 池
 -> 行情同步
 -> 行情清洗
 -> 数据质量检查
 -> 因子计算
 -> 策略运行
 -> 目标组合
 -> 当前持仓
 -> 持仓分析 / 定投建议 / 风控调仓
 -> 回测 / 报告 / AI 投研
```

AI 投研数据流：

```text
用户输入 ETF 代码
 -> 可选自动补全资产主数据和最近行情
 -> 读取 ETF 详情、因子、持仓、目标组合、持仓分析
 -> 规则型 Agent 生成观点
 -> DeepSeek 生成中文综合结论
 -> 保存到 agent_analysis_log
 -> 前端展示并可历史回看
```

## 部署结构

服务器路径：

```text
/opt/ETFSystematicPortfolioPlatform
```

部署方式：

```bash
git pull --ff-only
python3 scripts/deploy_production.py --compose-file compose.production.external-caddy.yml --env-file .env.production
```

部署脚本会：

1. 拉取最新代码。
2. 备份数据库。
3. 检查 Compose 配置。
4. 构建并启动容器。
5. 执行 Alembic 迁移。
6. 检查 API ready。

## 重要原则

- API Key 只放 `.env` 或 `.env.production`，不能提交 Git。
- 数据库结构变化必须写 Alembic migration。
- 前端新增接口必须同步更新 `frontend/src/api/client.ts`。
- 新增版本必须更新开发日志。
- 自动交易能力未来必须单独设计，不要混进现有建议逻辑。
