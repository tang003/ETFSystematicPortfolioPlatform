# 00 文档导航

本文档是 ETF Systematic Portfolio Platform 的文档入口。后续自己维护项目，或把项目交给 AI 修改时，建议先提供本文件、`10_系统流程说明.md` 和相关专题文档。

## 推荐阅读顺序

### 快速理解项目

1. `README.md`：项目概览、启动方式、主要功能。
2. `docs/01_项目说明.md`：项目定位、边界、当前能力和使用路径。
3. `docs/10_系统流程说明.md`：从业务流程角度理解系统如何运转。
4. `docs/09_开发日志.md`：了解每个版本做了什么。

### 准备修改后端

1. `docs/02_系统架构.md`：理解后端分层和数据流。
2. `docs/03_数据库设计.md`：确认表结构和字段含义。
3. `docs/04_API接口文档.md`：确认接口路径、请求参数和响应结构。
4. 对应业务文档，例如 `05_策略设计.md`、`06_风控设计.md`、`07_回测设计.md`。

### 准备修改前端

1. `README.md`：确认本地启动方式。
2. `docs/10_系统流程说明.md`：确认页面和业务流程关系。
3. `docs/04_API接口文档.md`：确认前端需要调用的 API。
4. `docs/09_开发日志.md`：确认最近版本是否改过相关页面。

### 准备部署或排查

1. `docs/08_部署说明.md`：查看 Docker、本地后端、本地前端启动方式。
2. `docs/10_系统流程说明.md`：查看常见排查路径。
3. `docs/09_开发日志.md`：确认最近版本的已知限制。

## 给 AI 的最小上下文包

如果未来让 AI 修改项目，建议至少提供：

- `README.md`
- `docs/00_文档导航.md`
- `docs/10_系统流程说明.md`
- `docs/09_开发日志.md`
- 和本次任务相关的专题文档，例如 API、数据库、策略、风控或回测文档

如果任务涉及代码，还应让 AI 先阅读相关目录：

- 后端接口：`backend/app/api`
- 后端业务：`backend/app/services`
- 数据模型：`backend/app/models`
- 请求响应模型：`backend/app/schemas`
- 前端页面：`frontend/src/views`
- 前端 API 客户端：`frontend/src/api/client.ts`

## 版本维护规则

每次完成一个可验收版本后，需要同步做四件事：

1. 更新相关设计文档。
2. 在 `docs/09_开发日志.md` 增加版本记录。
3. 本地跑必要测试和构建。
4. 提交 Git 并推送到 GitHub。

建议版本命名格式：

```text
v主版本.次版本.修订号-阶段说明
```

示例：

```text
v0.13.0-holdings-analysis
v0.13.1-docs-polish
```

## 当前关键限制

- 当前不连接真实券商账户。
- 当前不自动下单。
- 调仓单是建议单，需要人工确认。
- Tushare 等真实行情 API 暂未接入。
- 当前持仓由用户手动录入。
- 定投计划分析尚未实现。
- 登录、权限、多账户功能尚未实现。

## 修改文档时的要求

- 文档必须与当前代码状态一致，不能保留明显过期描述。
- 新增 API 时必须补 `04_API接口文档.md`。
- 新增数据表或字段时必须补 `03_数据库设计.md`。
- 新增业务流程时必须补 `10_系统流程说明.md`。
- 每次版本完成后必须补 `09_开发日志.md`。
- 中文为主，必要英文术语保留在括号中，例如 `Alpha Ranking`、`Target Portfolio`、`Rebalance Orders`。
